# story-first 模式

这个模式用于解决“词表牵着剧情走”的问题：先写正常英文故事，再从故事里回收红宝书词汇。

## 核心流程

1. 先写一篇自然英文故事草稿。
2. 用 `tools/story_first_pick.py` 扫描草稿。
3. 只保留红宝书里存在、且历史没有用过的词。
4. 同一词头只保留第一次命中，保证去重。
5. 支持常见词形还原，例如 `reduced` 计入 `reduce`，`stories` 计入 `story`，`wearing` 计入 `wear`。
6. 如果不足 200 个，不允许硬塞单词；应该扩写故事场景，再重新扫描。
7. 确定 200 个词后，再把这些词第一次出现的位置加粗，并生成后续词根、发音、例句、词源和资料依据。

## 写故事草稿的要求

- 先追求正常文章感：人物动机、环境、对话、动作、情绪推进要自然。
- 不要把单词写成菜单、日志、关键词清单、屏幕列表或路线提示。
- 英文故事可以分幕写，但每一幕都必须有真实事件：约会、误会、调查、共同修理、雨天共伞、吃饭、争执、和解等。
- 中文翻译可以后置；扫描阶段优先识别 `EN:` 行。如果没有 `EN:` 行，就扫描整篇英文草稿。
- 目标词不是预先指定的，而是从故事里实际出现的红宝书未用词中回收。

## 命令示例

```powershell
python "G:\我的云端硬盘\考研学习\Math_study\单词背诵\tools\story_first_pick.py" `
  --story-draft "G:\我的云端硬盘\考研学习\Math_study\单词背诵\每日任务\Day003_story-first英文草稿.md" `
  --theme "命运石之门：雨天共伞路线" `
  --count 200
```

如果草稿暂时不足 200 个红宝书未用词，脚本会停止并提示命中数量。此时应该扩写故事，而不是强行从词表补词。

## 与 ordered 模式的区别

- `ordered`：按红宝书上、中、下顺序抽取，进度最稳定，但剧情容易被词表牵着走。
- `story-first`：按故事中出现的红宝书词回收，文章最自然，但需要扩写到足够覆盖 200 个未用词。
